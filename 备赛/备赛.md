### 3. 将“细节”抽象为“不变量”或“个人模板”

算法题的细节确实多（比如边界条件、 $+1$ 还是 $-1$、开闭区间），这些细节如果在考场上靠猜，不仅极易出错，而且非常消耗心态。

- **确立唯一信仰（循环不变量）：** 以二分查找为例，不要去死记 `while(left <= right)` 还是 `while(left < right)`。选定一种你最舒服的区间定义（比如左闭右闭 `[left, right]`），然后所有的逻辑（包括 `right = mid - 1`，以及循环结束条件）都严格遵循这个区间定义。
    
- **总结个人易错点：** 建立一个轻量级的错题本，不需要记录完整的题目，只记录**你为什么会错这个细节**。例如：“注意：在拓扑排序中，入度为 0 的点入队时，如果是统计路径数量，要初始化起点的值为 1”。

### 2. 剥离代码，用“费曼技巧”复习思路

既然思路是最重要的，复习的检验标准就不应该是“能否默写出代码”，而是“能否流畅地表达出解题逻辑”。

- **三句话总结法：** 尝试用简练的自然语言把题目的思路说出来。例如复习“LRU 缓存机制”的思路：“本质是一个哈希表加双向链表。哈希表提供 $O(1)$ 的查询，双向链表提供 $O(1)$ 的节点移动。每次访问节点就把它移到头部，容量满了就淘汰尾部。”
    
- **画图模拟状态流转：** 尤其是动态规划、回溯算法和图论。不要看着代码回忆，而是拿出一张纸，自己画出状态树的推导过程或者 DP table 的填表顺序。如果图能画出来，代码只是水到渠成的事情。

**吃透母题的灵魂：** 对于母题，不仅要知道怎么做，还要问自己三个问题：

- 为什么这道题能用这个算法？（例如：为什么能用二分？因为具有单调性或两段性）。
    
- 它的核心瓶颈在哪里？（例如：寻找数组中满足条件的子串，暴力解法是 $O(N^2)$，滑动窗口是如何将其优化到 $O(N)$ 的？）。
    
- 它的通用模板是什么？

### 黑盒法 / 存根开发 (Stubbing & Top-Down Design)

这和填注释非常像，只不过你填的是“空函数”。当你面对一个复杂的算法题时，不要一上来就写细节，而是先假设你已经有了一些完美的工具函数（黑盒），先把主逻辑写出来。

- **怎么做**：比如写快速排序，你可以先写一个空的 `int partition(int[] arr, int L, int R) { return 0; }`。然后在 `main` 函数里把递归的大逻辑写完。主逻辑跑通没报错（虽然结果不对），你再钻进 `partition` 这个黑盒里去实现交换逻辑。
    
- **为什么有用**：它强迫你先进行“顶层设计”，避免在写主框架时被底层的边界条件（比如指针怎么移动、+1 还是 -1）扰乱心智。

这是一个非常高级的、“关于学习本身”的好问题！你意识到了**“顿悟”是可遇不可求的，而我们需要一种稳定、可复现的方法。**

确实，人的直觉和精力是有限的，你不可能（也不应该）对每道题的每一行代码都产生这种深度的灵魂拷问，那样太累了。

既然“被动产生疑问”不可靠，那我们就**“主动制造疑问”**。我教你三个非常实用的“思维脚手架”，当你觉得“这题我好像懂了，但又没完全懂”的时候，直接套用这三招：

**1. 主动破坏法（“找茬”测试）** 既然你不知道问题在哪，那就故意把代码写错。

- **乱序：** “如果我把这段 `while` 放到那个 `if` 后面会怎样？”（就像咱们今天做的一样）。
    
- **改边界：** “这里是 `<`，如果我改成 `<=`，跑测试用例会报错吗？为什么这个用例会挂？”
    
- **删逻辑：** “如果我把最后维护单调性的那个 `while` 删掉，程序是跑得慢了（超时），还是答案错了（WA）？” 当你故意破坏代码并看到它崩溃时，你立刻就能明白那行代码存在的真正意义。
    

**2. 费曼自问法（强行解释）** 遮住老师的代码和解析，看着题目，假装你要给一个完全不懂这道题的同学讲课。 当你讲到某个地方，发现自己只能说出“反正这里就是要用个单调队列”，而说不出“为什么别的结构不行，非得是单调队列”时，**卡壳的地方，就是你该向自己提问的地方。**

**3. 警惕“反直觉”地带** 像你今天之所以能挖出这么深的逻辑，就是因为你敏锐地察觉到了**“老师的写法”和“我的直觉”不一致**。 以后做题时，只要你心里冒出这样一个念头：“哎？他怎么先干这个了？换作是我，我肯定先干那个啊……” **立刻停下来！这就是一座金矿！** 不要顺着老师的思路滑过去，一定要像今天这样，把“为什么我的直觉错了”盘清楚。
